# 产品 / 项目经理（PM）交付物规范

PM 的关键产物是**清晰的权限矩阵与可测验收标准**。模糊需求会被直接打回。

## 核心职责

1. 需求拆分：按资源 × 动作维度列权限矩阵草案
2. 数据范围假设：明确"谁能看到谁的数据"
3. 租户可配置项：哪些由平台统一 / 哪些由租户自定义
4. 验收标准：可观测、可复现、可自动化
5. 上线沟通：灰度节奏、客户通知

## 需求模板

```markdown
# [模块/功能] 需求说明

## 背景与目标
- 业务背景：一句话
- 目标：可量化（例：订单审批 TAT 从 2h 降到 30min）
- 非目标：明确不做什么

## 角色与场景
| 角色 | 租户内层级 | 关键场景 |
|------|------------|----------|
| 销售 | 部门级 | 创建订单 |
| 销售经理 | 部门级（含下级） | 审批本部门订单 |
| 财务 | 租户级 | 审批跨部门订单 |
| 平台客服 | 平台级 | 协助排障（只读） |

## 资源清单
- 订单 Order：主资源
- 订单审批记录 OrderApproval：子资源
- 订单附件 OrderAttachment：关联资源

## 权限矩阵草案
| 权限码 | 销售 | 销售经理 | 财务 | 平台客服 |
|--------|:----:|:--------:|:----:|:--------:|
| crm:order:read | ✅(Self) | ✅(Dept) | ✅(Tenant) | ✅(Global-R) |
| crm:order:create | ✅ | ✅ | - | - |
| crm:order:approve | - | ✅(Dept, amount≤100k) | ✅(Tenant) | - |
| crm:order:export | - | ✅(Dept) | ✅(Tenant) | - |

## 数据范围
- 默认：按创建人/部门
- 例外：财务可见全租户
- 导出：复用列表范围

## 字段敏感级别
| 字段 | 敏感级 | 默认策略 |
|------|--------|----------|
| customerPhone | L2 | PartialMask |
| bankAccount | L3 | Hash（详情可申请原值） |

## ABAC 条件（可选）
- 销售经理审批金额上限由租户配置（默认 10 万）
- 仅工作日 09:00-21:00 可发起审批（可租户级开关）

## 租户可配置项
| 配置键 | 类型 | 默认 | 可见范围 |
|--------|------|------|----------|
| crm.approve.amount_limit | number | 100000 | 租户管理员 |
| crm.approve.working_hours_only | bool | false | 租户管理员 |

## 验收标准
- 每条权限矩阵单元对应一个 E2E 用例，自动化
- 跨租户访问 100% 拒绝
- 敏感字段在导出文件中已脱敏（抽查 10 份）
- 审批 p95 响应 < 500ms
- 权限变更 2s 内全量生效

## 上线计划
- 灰度节奏：详见 DevOps 发布方案
- 客户通知：提前 7 天邮件 + 产品公告
- 回滚触发线：授权失败率 > +1% / 跨租户告警 > 0
```

## 权限矩阵填写约定

- 单元格内容：`✅`（允许）/ `-`（拒绝）
- 带范围：`✅(Self)` `✅(Dept)` `✅(Tenant)` `✅(Global)` `✅(Global-R)` 等
- 带条件：`✅(Dept, amount≤100k)` — 条件用自然语言简写，对应 ABAC 策略 ID 在附录维护

## 与架构师 / 安全的协作

PM 的权限矩阵是架构师 ADR 的**输入**，也是安全越权用例的**输入**：

- 架构师据此选择隔离模式与扩展点
- 安全据此生成"谁不能访问谁"的用例
- 测试据此生成权限矩阵测试用例

## 验收标准写法

**坏**："权限应当正确"
**好**：

```
- 销售 A 访问销售 B 的订单详情 → 返回 403
- 销售经理 M 查询列表 → 仅返回自己部门订单（含下级部门），不包含其他部门
- 财务 F 导出订单 → Excel 中 customerPhone 字段为 `138****5678` 格式
- 平台客服 CS 跨租户查询 → 返回 200 但不可写操作（尝试 PUT/DELETE → 403）
```

## 变更管理

权限矩阵变更需要：
- 新旧矩阵 diff
- 影响的租户/用户清单
- 回滚方式
- 客户沟通稿（是否影响用户已有操作）

## PM 评审清单

```
- [ ] 角色清单完整，含平台侧与租户侧
- [ ] 权限矩阵覆盖所有资源 × 动作
- [ ] 数据范围对每个权限明确标注
- [ ] 敏感字段与默认策略已列
- [ ] ABAC 条件有对应业务语义
- [ ] 租户可配置项与默认值已明
- [ ] 验收标准可自动化
- [ ] 灰度节奏与回滚触发线写入
- [ ] 客户沟通稿已起草
```

## 常见 PM 反模式

- ❌ 只写"普通用户/管理员"两级，权限矩阵一片模糊
- ❌ "有权限就能做" — 未区分 Self / Dept / Tenant
- ❌ 把"平台超管"与"租户管理员"混为一谈
- ❌ 导出 / 打印功能在需求里被遗漏，上线后成为合规事故源
- ❌ 验收标准是"客户觉得好用"
- ❌ 需求变更口头传达，不更新矩阵与 ADR
