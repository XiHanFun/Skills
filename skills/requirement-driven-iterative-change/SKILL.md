# 需求驱动、分步改代码

## 何时启用

用户在描述功能、修复或重构时，期望 **先列清需求项**、**按项改代码**，且结果 **可追溯、可回滚、可验证** 时，按本流程执行。

## 阶段一 — 先理解现有代码

在动手改代码前：

1. **定位** 相关文件（搜索、打开邻域、读调用处）。
2. **归纳** 当前行为与约束，用 3～8 条要点（来自代码事实，不臆测）。
3. 仍有不确定处则 **继续阅读** 或 **只问一个具体问题**，不虚构 API。

## 阶段二 — 拆解需求（必须做）

输出带稳定编号的列表，本会话内固定使用：

| 字段 | 规则 |
|------|------|
| ID | `REQ-001`、`REQ-002`…（递增） |
| 标题 | 简短动宾短语 |
| 范围 | 可能涉及的文件/模块（尽力而为） |
| 完成标准 | 可观察的验收条件（可测或可感知） |
| 依赖 | 可选：依赖的 `REQ-00x` |

单项应 **小到适合一次提交或一批逻辑相关的编辑**。模糊需求拆成多条 REQ。

**模板（复制后填写）：**

```markdown
## 需求项

| ID | 标题 | 范围 | 完成标准 | 依赖 |
|----|------|------|----------|------|
| REQ-001 | … | … | … | — |
```

## 阶段三 — 按项实现

按顺序处理每个 `REQ-00n`（遵守依赖关系）：

1. 明确写出：**正在实现 REQ-00n：[标题]**
2. **只** 做该 REQ 需要的改动，避免范围蔓延。
3. **可追溯**（按仓库习惯选）：
   - **任务列表**：每项 REQ 对应一条 todo，完成即勾选。
   - **提交说明**：若用户用 git，主题行包含 `REQ-00n`。
   - **代码注释**：仅在行为与 REQ 对应关系不明显时简短标注。

## 阶段四 — 回滚点

优先保证 **可逆**，不依赖用户记忆：

1. **Git**：若仓库使用 git，建议 **每个 REQ 一次提交**（或一批原子变更）。回滚用 `git revert <commit>` 或在分支上回到上一提交。
2. **分支**：较大改动建议 **功能分支**，主干保持安全；全部 REQ 验证通过后再合并。
3. **无 git**：在回复中列出 **每个 REQ 改动的文件**，便于用户用备份或 IDE 本地历史恢复。

## 阶段五 — 每项完成后验证

每完成一条需求项，在继续下一条前运行 **与项目匹配** 的检查：

- **测试**：仓库文档中的单测/集成命令（如 `dotnet test`、`pnpm test` 等）。
- **Lint / 类型检查**：如 `eslint`、`tsc --noEmit` 或解决方案构建，与技术栈一致。
- **手工**：仅当自动化无法覆盖时；写明具体操作场景。

若验证失败，在 **同一 REQ 内修复**，或拆出新的 `REQ-00m` 跟进——不要留下未声明的坏状态。

## 阶段六 — 收尾

全部 REQ 完成后：

1. **汇总表**：REQ ID → 改动（文件或区域）→ 如何验证。
2. **回滚提示**：若使用 git，给出分支名或提交范围 / revert 命令。

## 反模式

- 不列需求表直接写代码。
- 多条 REQ 混在同一坨 diff 里无法区分。
- REQ 标完成却没有任何验证步骤。
- 大块重构不绑定具体 REQ ID。
